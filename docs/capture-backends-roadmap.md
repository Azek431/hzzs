# 截图方式取舍与后续路线

> **状态**：规划文档
>
> 本文档分析 HZZS 项目未来屏幕采集模块的各种实现方案，包括取舍分析和路线规划。

## 1. 需求概述

视觉识别模块需要获取游戏画面的每一帧像素数据，输入给颜色分割/模板匹配等识别算法。截图方式的选择直接影响：

- **延迟**：从画面渲染到算法处理的端到端延迟
- **性能开销**：CPU/GPU/内存占用
- **合规性**：是否违反应用商店政策或游戏规则
- **兼容性**：支持的 Android 版本和设备范围
- **稳定性**：长时间运行的可靠性

当前项目的帧间隔目标为 50ms（20fps HUD 渲染），截图延迟需控制在 100ms 以内才能保证 ETA 提示的有效性。

## 2. 可选方案对比

### 2.1 MediaProjection（系统级截屏 API）

**方案描述**：使用 Android 系统的 `MediaProjection` API + `ImageReader` 获取屏幕像素数据。

| 维度 | 评估 |
| --- | --- |
| 延迟 | 30~80ms（取决于刷新率和缓冲） |
| 性能开销 | 中（GPU 合成层截图，不影响游戏渲染） |
| 兼容性 | API 21+（本项目 minSdk=24，完全覆盖） |
| 用户授权 | 需要系统截屏授权对话框（一次性） |
| 合规性 | 良好（系统 API，Google Play 允许） |
| 实现复杂度 | 低（Android 官方 API，文档完善） |
| 后台能力 | 有限（应用退出后截屏停止，需要前台服务维持） |
| 横竖屏 | 需要处理屏幕旋转和方向变化 |

**优点**：
- 官方 API，稳定性最好
- 不需要 Root 或特殊权限
- Google Play 上架无障碍
- 与 `ForegroundService` 配合可维持后台截屏

**缺点**：
- 需要用户手动授权（首次弹出系统对话框）
- 下拉通知栏有截屏指示（可能引起用户困惑）
- 部分厂商 ROM 可能限制后台 `MediaProjection`
- 无法截取加密内容（DRM 保护的画面）

**推荐度**：高。作为 **Milestone 2 的首选方案**。

---

### 2.2 Shizuku / ADB 无线截屏

**方案描述**：通过 Shizuku（无需 Root 的 ADB 权限框架）或传统 ADB 执行 `screencap` 命令获取屏幕截图。

| 维度 | 评估 |
| --- | --- |
| 延迟 | 50~150ms（进程启动开销大） |
| 性能开销 | 高（每次调用启动新进程） |
| 兼容性 | 需要 ADB 调试权限 |
| 用户授权 | 需要 USB 连接授权或无线调试配对 |
| 合规性 | 一般（非正式分发渠道） |
| 实现复杂度 | 中 |
| 后台能力 | 差（不适合持续运行） |

**优点**：
- 不需要 Root
- 适合开发和调试阶段

**缺点**：
- 每次调用 `screencap` 需要 fork 新进程，开销极大
- 无法实现持续帧流（只能单帧截图）
- 需要开发者选项中的 USB 调试/无线调试权限
- 不适合作为最终用户的发行方案

**推荐度**：低。仅适合 **开发调试阶段** 的临时方案。

---

### 2.3 Root 权限截屏

**方案描述**：通过 Root 权限直接读取 `/dev/graphics/fb0` 或使用 `su` 执行 `screencap`。

| 维度 | 评估 |
| --- | --- |
| 延迟 | 10~30ms（最低） |
| 性能开销 | 低（直接内存访问） |
| 兼容性 | 仅限 Root 设备 |
| 用户授权 | 需要用户主动 Root 设备 |
| 合规性 | 差（违反 Google Play 政策，可能触发 SafetyNet） |
| 实现复杂度 | 中 |
| 后台能力 | 好（Root 权限不受前台服务限制） |

**优点**：
- 延迟最低
- 可以获取底层帧缓冲，不受系统限制
- 可以截取加密/DRM 内容

**缺点**：
- 需要用户 Root 设备，门槛极高
- 违反 Google Play 政策，无法上架
- 触发 SafetyNet/Play Integrity，可能影响游戏运行
- 用户信任成本高

**推荐度**：极低。仅在 **内部测试** 或 **非分发场景** 下考虑。

---

### 2.4 虚拟屏幕 / 悬浮窗叠加

**方案描述**：通过 `SYSTEM_ALERT_WINDOW` 在屏幕上绘制虚拟覆盖层，间接获取画面信息（如像素采样）。

| 维度 | 评估 |
| --- | --- |
| 延迟 | 取决于实现方式 |
| 性能开销 | 中到高 |
| 兼容性 | 需要 SYSTEM_ALERT_WINDOW 权限 |
| 用户授权 | 需要悬浮窗权限（本项目已申请） |
| 合规性 | 一般（悬浮窗本身合规，但像素采样可能有争议） |
| 实现复杂度 | 高 |
| 后台能力 | 取决于悬浮窗存活 |

**优点**：
- 利用已有的悬浮窗权限体系
- 不需要额外的 Root 或 ADB 权限

**缺点**：
- 像素采样效率低（需要从 View 逐像素读取）
- 悬浮窗本身会影响游戏画面
- 部分设备不支持从悬浮窗读取像素
- 实现复杂度高，收益低

**推荐度**：低。技术上可行但工程性价比不高。

---

### 2.5 无障碍服务辅助截图

**方案描述**：利用 `AccessibilityService` 的 `getWindows()` 获取 `AccessibilityWindow` 信息，结合 `performGlobalAction(GLOBAL_ACTION_SCREENSHOT)` 触发截屏。

| 维度 | 评估 |
| --- | --- |
| 延迟 | 200~500ms（太高） |
| 性能开销 | 中 |
| 兼容性 | 需要无障碍服务权限 |
| 用户授权 | 需要无障碍服务授权 |
| 合规性 | 有风险（自动截屏可能被判定为滥用） |
| 实现复杂度 | 中 |
| 后台能力 | 好（无障碍服务可后台运行） |

**优点**：
- 可利用已有的无障碍服务权限
- 不需要额外授权

**缺点**：
- 延迟过高，无法满足实时 HUD 需求
- `GLOBAL_ACTION_SCREENSHOT` 是全局操作，影响用户体验
- 无障碍服务主要用于 UI 信息获取，非设计用途
- 合规风险较高

**推荐度**：低。延迟和合规性都不达标。

---

## 3. 推荐路线

### Milestone 2（当前阶段）：MediaProjection + ImageReader

**理由**：
1. 延迟和性能在可接受范围内（30~80ms）
2. 官方 API，稳定性有保障
3. Google Play 上架无障碍
4. 实现复杂度低，可以快速验证视觉识别管线

**实现要点**：
```
MediaProjectionManager.createScreenCaptureIntent()
    → 用户授权 → onActivityResult(intent)
    → MediaProjection.attach(ImageReader)
    → ImageReader.acquireLatestImage() → 每帧像素数据
    → 颜色分割/模板匹配 → FrameDetections
    → NativeAnalysisEngine.Analyze() → AnalysisResult
    → HUDCanvasView 渲染
```

**注意事项**：
- 使用 `ImageFormat.RGB_888` 避免色彩空间转换开销
- `ImageReader` 的缓冲队列长度设为 2（减少延迟）
- 分辨率适配：按目标游戏分辨率缩放，而非全屏幕分辨率
- 需要前台服务维持 `MediaProjection` 存活

### Milestone 3（优化阶段）：MediaProjection + GPU 加速

**改进方向**：
1. 使用 `SurfaceTexture` + OpenGL ES 直接在 GPU 上处理帧数据，避免 CPU-GPU 数据拷贝
2. 使用 RenderScript 或 NNAPI 加速颜色分割
3. 分辨率自适应：根据设备 DPI 自动调整处理分辨率

### Milestone 4（高级阶段）：Shizuku 备用方案

**适用场景**：
- 部分厂商 ROM 限制 `MediaProjection` 后台运行时
- 作为降级方案，通过 Shizuku 提供"用户主动触发的截屏"模式

### 不推荐的方案

| 方案 | 不推荐原因 |
| --- | --- |
| Root 截屏 | 用户门槛过高，违反分发政策 |
| 无障碍截屏 | 延迟过高，合规风险 |
| 虚拟屏幕 | 工程性价比低 |
| ADB screencap | 仅适合调试，不适合产品 |

## 4. 测试方法

### 4.1 延迟测试

| 测试项 | 方法 | 合格标准 |
| --- | --- | --- |
| 截图延迟 | 使用外部相机拍摄屏幕，对比画面渲染时间与截图获取时间 | < 100ms |
| 端到端延迟 | 从画面变化到 HUD 提示显示的时间差 | < 200ms |
| 帧率稳定性 | 连续运行 30 秒，统计帧间隔的标准差 | 标准差 < 5ms |

### 4.2 性能测试

| 测试项 | 方法 | 合格标准 |
| --- | --- | --- |
| CPU 占用 | 通过 `dumpsys cpuinfo` 监控应用 CPU 使用率 | < 15% |
| 内存占用 | 通过 `dumpsys meminfo` 监控 RSS | < 200MB |
| 电池消耗 | 连续运行 1 小时的电量消耗 | < 10% |
| 温度 | 设备背面温度变化 | < 5 度 |

### 4.3 兼容性测试

| 测试维度 | 覆盖要求 |
| --- | --- |
| Android 版本 | API 24 ~ API 34（全覆盖） |
| 屏幕分辨率 | 至少覆盖 570x1280、1080x2400、1440x3200 三个档位 |
| 厂商 ROM | 小米、华为、OPPO、vivo、三星各至少一台 |
| 折叠屏 | 测试展开/折叠状态下的截图适配 |

## 5. 已知风险

1. **厂商 ROM 限制**：部分厂商（尤其是小米 MIUI、华为 EMUI）会限制后台服务，可能导致 `MediaProjection` 在应用退到后台后被系统杀死。需要通过前台通知服务 + 白名单引导缓解。

2. **截屏指示**：Android 10+ 在截屏时会在状态栏显示通知。虽然 `MediaProjection` 的通知不可关闭，但可以通过前台服务的通知合并减少用户困惑。

3. **分辨率缩放精度**：将高分辨率屏幕缩放到较低分辨率处理后，可能丢失小物体的细节（如远处的毒瓶）。需要在性能和精度之间权衡。

4. **横屏适配**：如果游戏支持横屏，截图方向需要正确处理。`MediaProjection` 返回的图像方向与设备传感器方向相关，需要动态适配。

5. **多窗口模式**：在分屏或多窗口模式下，截图区域可能不是完整屏幕。需要通过 `DisplayCutout` 和 `WindowInsets` 精确计算游戏可视区域。

6. **DRM 内容**：部分游戏版本可能在特定画面（如登录界面）使用 DRM 保护，`MediaProjection` 会返回黑屏。需要在代码中检测并优雅降级。
